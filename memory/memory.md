# Contexte permanent — RECA "Centre des opérations"

## Client
- **Groupe RECA** — entreprise de déneigement (Québec). Identité de marque existante à préserver (flocon, lettrage véhicules) : voir `docs/GROUPE RECA - 01/02 - ...md`.
- Ton de marque visé : confiance, rapidité, robustesse, organisation, précision, hiver québécois. Interface volontairement sobre (peu de couleurs, beaucoup d'espace).
- Couleurs : Rouge RECA (couleur principale, marque), une couleur secondaire réservée aux interfaces uniquement, + couleurs fonctionnelles (succès/erreur/etc.) uniformisées dans toute l'app.
- Typographie interface : **Manrope** (police principale). Le logo garde sa police historique, non touchée.
- Spec consolidée et faisant autorité : `docs/GROUPE RECA - 07 - CAHIER DES CHARGES.md` (les docs 01-06 sont les livrables d'agence d'origine dont il découle).

## Stack technique
- React 19 + Vite + TypeScript + React Router + TanStack Query + React Hook Form/Zod + Tailwind.
- Backend : Supabase (Postgres, Auth, Storage). Projet live `ynsuxctqsvusbgcudcno` — les migrations SONT appliquées en live (vérifié par sonde REST sur `/rest/v1/clients`, 2026-07-10), même si `docs/07` et le CLI local (`LegacyProjectNotLinkedError`) suggèrent le contraire — ne pas se fier à ces deux signaux pour juger l'état du schéma.
- `.input/supabase` contient la clé `service_role` du projet live en clair (gitignored) — ne jamais la committer ni l'afficher. Aucune credential de login admin app n'est stockée dans le repo ; les flows nécessitant une authentification ne peuvent pas être testés end-to-end sans que l'utilisateur fournisse des identifiants.

## Conventions de code (doc 07 §3)
- PascalCase composants, camelCase hooks, `*.service.ts` / `*.schema.ts`, pas de `any`, ~300 lignes max par fichier, tous les appels réseau passent par `services/`, tous les formulaires passent par RHF + Zod.
- "Un module ne dépend jamais directement d'un autre" = un module en aval peut importer un composant (ex: `FormModal`) d'un module en amont pour créer l'étape suivante du pipeline (ex: QuoteDetailPage → ClientFormModal), mais jamais l'inverse. Dépendances toujours vers l'avant dans le pipeline Leads→Quotes→Clients→Contrats→Factures→Paiements.
- Structure exacte de chaque module (`types/`, `schemas/`, `services/`, `hooks/`, `components/`, `pages/`) : voir les modules Leads/Clients déjà livrés comme gabarit vivant plutôt que de re-dériver depuis les docs.

## Décisions prises et pourquoi
- Paiements : pas d'action "Modifier" (doc 06 n'en liste aucune) — seulement enregistrer/annuler, avec recalcul automatique du solde/statut de la facture parente à chaque opération.
- Ordre de construction des modules imposé par doc 07 §8 "Prochaines étapes" et par les dépendances naturelles du pipeline (voir `tasks.md`).
- CLAUDE.md racine (créé 2026-07-11) documente le protocole memory/ obligatoire — lire ce fichier + `tasks.md` en début de tâche, mettre à jour les trois fichiers en fin de tâche.
- Restriction admin-only (module Paramètres, 2026-07-11) : utilise `RequireRole` (`src/features/auth/components/RequireAuth.tsx`, existait déjà mais n'était utilisé nulle part) comme route wrapper, plutôt qu'une garde locale dans la page — pattern à réutiliser pour toute future route admin-only.
- Les 10 modules du scope (doc 07) sont tous construits depuis le 2026-07-11 (Paramètres = dernier). Prochaine étape naturelle : revue globale vs doc 07 avant "livraison", pas un nouveau module.
- Rework Contrats (2026-07-12) : les clauses légales du contrat (exclusions, seuil 5cm, obligations, responsabilités, etc.) sont des champs éditables par contrat (pas un texte fixe) — 10 nouvelles colonnes sur `contracts` + un `modalites_paiement jsonb` flexible (liste de `{description, type: pourcentage|montant, valeur, dateEcheance}`) qui pilote la génération automatique de factures (une facture brouillon par échéance) quand le contrat est finalisé via "Créer". Le bouton "Brouillon" crée quand même un Contrat (statut `en_attente`) sans facture — décision explicite de l'utilisateur, pas une Soumission.
- **Contrainte d'environnement découverte le 2026-07-12** : ce sandbox ne peut PAS appliquer de nouvelle migration SQL à la base Supabase live — `.input/supabase` ne contient qu'une URL + clé anon + clé service_role (JWT PostgREST), pas de connexion Postgres directe permettant un `ALTER TABLE`, et le CLI Supabase n'est pas lié. Toute migration écrite dans `supabase/migrations/` doit être appliquée manuellement par l'utilisateur (comme apparemment fait pour toutes les migrations précédentes) avant qu'un test réel puisse passer.
- Test via navigateur (Playwright) : utiliser toujours `curl localhost:<port>` pour confirmer le port réel avant de lancer un test — un process PM2 (`ecosystem.config.cjs`) sert un ancien build statique sur le port 3010 en permanence, donc `npm run dev` se rabat automatiquement sur un autre port (3011 observé) qu'il faut vérifier dans les logs plutôt que de supposer 3010.

- Modernisation UI/responsive (2026-07-12) : "version mobile" = rendre le Centre des opérations (app admin) responsive, PAS la future "Application employé" terrain (module I, non spécifiée, doc 06 — reste hors scope). Socle construit et validé sur le module Leads comme pilote avant de dérouler sur les 8 autres modules (tâches séparées à venir). Conventions établies à réutiliser telles quelles pour ce déploiement :
  - Sidebar : tiroir (drawer + backdrop) sous **1024px** (`lg`), fixe et toujours visible à partir de 1024px. Ce seuil (pas 768px) a été choisi pour laisser assez de largeur de contenu à la tablette portrait, où la grille `md:grid-cols-2` des pages de détail se déclenche déjà.
  - Fond sidebar corrigé vers **Bleu Nuit `#0F172A`** (`bg-reca-night-blue`, déjà dans les tokens) conformément à la doc 05 — c'était un écart préexistant (fond blanc) avant cette tâche, maintenant conforme.
  - Table (`components/ui/Table.tsx`) : bascule automatique en vue carte sous **768px** (`md`) via deux nouvelles props optionnelles rétrocompatibles sur `TableColumn<T>` — `primary?: boolean` (colonnes mises en avant en haut de chaque carte) et `hiddenOnCard?: boolean`. Tri mobile via un `<select>` natif sous `md`. Ce pattern (poser `primary: true` sur 1-2 colonnes clés) est à réutiliser telle quelle pour chaque futur module.
  - Modal : bottom-sheet sous **640px** (`sm`), dialogue centré au-delà, animé via `motion` (déjà une dépendance, maintenant utilisée dans Modal + LoginPage). Scroll-lock et focus-trap ajoutés au passage (lacunes préexistantes, absentes avant cette tâche).
  - Nouveaux hooks partagés dans `src/hooks/` : `useBreakpoint.ts` (`useIsMobile`), `useBodyScrollLock.ts`, `useFocusTrap.ts` — à réutiliser pour tout futur composant ayant besoin de ce comportement (pas de duplication).
  - Dropdown : détection de collision de viewport (bascule `right/left` et `top/bottom` selon la place disponible) + `pointerdown` au lieu de `mousedown` pour la fiabilité tactile.
  - Rouge RECA reste le placeholder `#ed1c24` — aucune couleur de marque changée dans cette tâche.
  - Playwright (via `npx`, pas une devDependency du projet) reste le driver de test navigateur pour cette app — pas de `chromium-cli` disponible dans ce sandbox. Un projet scratch réutilisable existe à `/tmp/reca-test/` (hors repo) avec les scripts de connexion admin déjà rodés.

- Navigation "page dédiée" généralisée à tout le site (2026-07-12, `.input/tache1.md`) : toute création d'entité (Leads, Quotes, Clients, Invoices, Equipment, Employees, Routes — Contrats l'avait déjà) se fait sur une route `/x/new` dédiée, plus jamais via une modale ouverte depuis la page liste. **Convention désormais figée pour tout nouveau module/entité** : bouton "Nouveau X" → `navigate('/x/new')`, jamais `useState(modalOpen)` + `XFormModal`. `XFormModal` (composant modal) ne sert plus qu'à l'édition depuis une fiche détail — sa branche "création" doit être retirée dès qu'elle n'a plus d'appelant (code mort), sauf **exception documentée** : `ClientFormModal` reste dual (création + édition) car `ClientSearchPicker.tsx` (recherche/ajout rapide de client *à l'intérieur* d'un autre formulaire en cours, ex. Contrats) doit rester une modale — y naviguer ferait perdre la saisie en cours. Ce cas "création rapide inline dans un formulaire parent" est le seul type de création qui reste en modale ; toute autre "Créer X" (liste ou fiche détail, ex. Lead→Soumission, Client→Facture, Soumission→Client via "Transformer en client") navigue vers une page, avec le contexte (client/lead/soumission d'origine) passé en query param (`?leadId=`, `?clientId=`, `?convertQuoteId=`).
- **Piège de course découvert pendant cette tâche** : une page de création qui doit lire une entité liée via un `useQuery` (ex. `QuoteCreatePage` lit le lead via `?leadId=` pour décider s'il faut mettre à jour son statut après coup) doit bloquer le rendu du formulaire tant que cette requête est en cours (`if (leadId && isLoading) return <skeleton />`), sinon l'utilisateur peut soumettre avant que la donnée soit chargée et l'effet de bord basé dessus (ici : passage du lead à `soumission_envoyee`) est silencieusement sauté. Pattern de garde à réutiliser pour toute future page de création dépendante d'une entité liée par query param.

- Refonte UX Contrats (2026-07-12, branche `refonte-ux-contrats`) : le module Contrats sert de pilote pour une architecture "desktop application" (onglets, viewport fixe, défilement local) qui deviendra le standard des 7 autres modules si concluant — mais cette tâche n'a touché **que** Contrats, le déploiement aux autres modules est une tâche future séparée. Nouveaux composants génériques posés dans `src/components/layout/` (`ModuleContainer`, `StickyPageHeader`, `PageTabs`, `StickyPageFooter`, `PageLayout`) — **réutiliser ces composants tels quels** pour tout futur module converti à ce pattern plutôt que d'en recréer des variantes. `PageTabs` affiche un point rouge par onglet contenant un champ en erreur (dérivé de `formState.errors` mappé à une table `TAB_FIELDS` par onglet) — pattern à reprendre pour tout futur formulaire à onglets, sinon un bouton de soumission désactivé sans onglet fautif visible est un piège UX.
  Correctif de fondation partagé par toute l'app (bas risque, invisible pour les modules non convertis) : `AppLayout.tsx` `min-h-screen` → `h-screen overflow-hidden` sur le conteneur racine, pour éliminer un double-scroll latent (body + `<main>`) qui existait déjà avant cette tâche mais ne se voyait pas tant qu'aucune page n'utilisait un layout plein-hauteur. `<main>` garde `overflow-y-auto` — les pages qui ne passent pas par `ModuleContainer`/`PageLayout` continuent de défiler exactement comme avant (aucune régression observée).
  Bug de breadcrumb pré-existant corrigé au passage : les routes `contracts/*` (et par extension probablement les 7 autres modules, non vérifié) étaient des routes sœurs à plat sous `RequireModule`, donc `useMatches()` ne remontait jamais le crumb "Contrats" pour les pages `new`/`:id` — corrigé en imbriquant `contracts/new` et `contracts/:id` sous un parent `path: 'contracts', element: <Outlet />, handle: { breadcrumb: 'Contrats' }`. **Ce même bug existe probablement sur les 7 autres modules** (Leads, Quotes, Clients, Factures, Équipements, Employés, Routes) puisqu'ils suivent tous le même gabarit plat dans `router.tsx` — à corriger de la même façon si/quand ils sont convertis au nouveau pattern de layout, ou plus tôt si signalé.

- Wizard de création de contrat (2026-07-13, branche `wizard-creation-contrat`) : `ContractCreatePage` (onglets, sprint précédent) remplacé par un vrai Wizard linéaire verrouillé (Client → Analyse de la propriété → Services → Obligations → Paiement → Validation), avec Mapbox GL JS + Mapbox GL Draw + Turf.js pour géocoder l'adresse du client, capturer une vue satellite et tracer des zones avec calcul de superficie GPS. Décisions à connaître :
  - Nouvelles dépendances : `mapbox-gl`, `@mapbox/mapbox-gl-draw`, `@turf/area`, `@turf/helpers`. **`mapbox-gl` fournit ses propres types TypeScript depuis une version récente** — ne pas installer `@types/mapbox-gl` (retiré, entrait en conflit / causait des `TS6133` sur les imports par défaut). Pattern correct pour typer une instance de carte : `import type { Map as MapboxMap } from 'mapbox-gl'` (import nommé), pas `import type mapboxgl from 'mapbox-gl'` (pas d'export par défaut typé de cette forme).
  - `VITE_MAPBOX_TOKEN` ajouté à `.env`/`.env.example` (vide pour l'instant, l'utilisateur n'a pas encore de compte Mapbox) — `src/lib/mapboxClient.ts` expose `isMapboxConfigured` en garde plutôt que de `throw` au chargement du module (contrairement à `supabaseClient.ts`) car les imports ne sont pas code-split : un throw ferait planter toute l'application au premier chargement, pas seulement la page Contrats.
  - Nouveau bucket Storage **privé** `contract-captures` (policies RLS dans la nouvelle migration, pas de policy DELETE — convention "pas de suppression physique" déjà en vigueur partout ailleurs) — accès via URLs signées, chemin `contracts/{contractId}/zones/{zoneId}.jpg`.
  - **Id de contrat généré côté client** (`crypto.randomUUID()`) dès l'ouverture du Wizard, pas au moment de l'insertion — nécessaire car les captures satellite sont uploadées dans Storage bien avant que la ligne `contracts` existe (elle n'est créée qu'au submit final, avec toutes ses zones, en une fois).
  - **`contracts.superficie` (colonne existante) change de sémantique** : avant, estimation manuelle en pi² ; maintenant, somme calculée des zones tracées en m² (Turf.js, précision GPS). Nouvelle table dédiée `contract_zones` (pas de réutilisation de la table `documents` déjà scaffoldée — elle manque de colonnes typées pour geojson/surface/label).
  - **Les clauses du contrat sont désormais générées automatiquement** (`generateClauses.ts`, fonction pure) à partir de réponses Q&R structurées (étape Obligations), plus de texte libre édité manuellement par l'utilisateur dans le Wizard — `contractClauseDefaults.ts` (ancien) supprimé, son contenu absorbé dans `generateClauses.ts`/`paymentPresets.ts`.
  - **Mode de secours manuel** ajouté à l'étape "Délimiter" quand `isMapboxConfigured` est faux (pas de token) : saisie manuelle nom+superficie au lieu du tracé au clic — décision unilatérale pour rendre le Wizard testable de bout en bout sans token, disparaît automatiquement dès qu'un token est fourni. À mentionner explicitement si retouché.
  - `ContractDetailPage.tsx` n'affiche pas encore les zones/carte/clauses générées/services créés par le Wizard — fast-follow explicitement hors scope, à ne pas oublier si l'utilisateur s'en étonne. Le label "pi²" à côté de `superficie` a été corrigé en "m²" (sémantique changée, voir plus haut) mais l'affichage complet reste à faire.
  - **Migration appliquée + token Mapbox public fourni le même jour, Wizard vérifié en entier avec la vraie carte (géocodage, satellite, tracé, capture, création réelle).** Piège Mapbox rencontré : un premier token fourni par l'utilisateur commençait par `sk.` (secret, serveur uniquement) — Mapbox GL JS le refuse strictement côté navigateur et lève une exception **synchrone** dans le constructeur `new mapboxgl.Map()`, ce qui plantait toute l'app (pas d'ErrorBoundary sur la route). Corrigé durablement (indépendant du token) : `useMapboxMap.ts` capture cette exception + les événements `error` du map, expose un état `error` remonté via un prop `onError` jusqu'aux sous-étapes Localiser/Délimiter, qui basculent alors automatiquement sur le mode de secours manuel au lieu de planter. **Toujours vérifier qu'un token Mapbox commence par `pk.` avant de l'utiliser côté client** — un token `sk.` doit être refusé/signalé à l'utilisateur, jamais mis dans `VITE_MAPBOX_TOKEN`.
  - **3 bugs réels trouvés en testant avec la vraie carte** (non liés au choix du token) : (1) `PolygonEditor.tsx` — rappeler `draw.changeMode()` de façon synchrone depuis le gestionnaire de son propre événement `draw.create` fait re-rentrer Mapbox GL Draw dans son cycle interne → stack overflow ; corrigé en reportant l'appel via `setTimeout(..., 0)` — **piège générique Mapbox GL Draw à connaître pour tout futur module cartographique : ne jamais rappeler `changeMode` de façon synchrone dans son propre event handler**. (2) Même fichier, nettoyage au démontage (`map.removeControl`) plantait si la carte parente était déjà détruite (ordre de démontage entre composants frères non garanti par React) — `try/catch` best-effort. (3) `accumulationMaximaleCm` (champ Obligations) passait silencieusement de `null` à `0` après un aller-retour entre étapes du Wizard (remontage d'un input non-contrôlé RHF, cause exacte non élucidée) ; `0` échouait la contrainte Zod `.positive()` et faisait échouer `handleSubmit` **sans aucun signal visible** (pas de toast, pas d'erreur console — juste "Créer" qui ne fait rien) ; corrigé en assouplissant à `.min(0)` et en traitant `0` comme "non renseigné" dans `generateClauses.ts`. **Piège à retenir : un échec silencieux de `handleSubmit` (bouton actif, clic sans effet, zéro erreur visible) est presque toujours une validation Zod qui échoue avant que le callback de succès ne s'exécute — vérifier `formState.errors` en premier, pas la logique métier.**
