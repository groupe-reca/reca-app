# Contexte permanent — RECA "Centre des opérations"

## Client
- **Groupe RECA** — entreprise de déneigement (Québec). Identité de marque existante à préserver (flocon, lettrage véhicules) : voir `docs/GROUPE RECA - 01/02 - ...md`.
- Ton de marque visé : confiance, rapidité, robustesse, organisation, précision, hiver québécois. Interface volontairement sobre (peu de couleurs, beaucoup d'espace).
- Couleurs : Rouge RECA (couleur principale, marque), une couleur secondaire réservée aux interfaces uniquement, + couleurs fonctionnelles (succès/erreur/etc.) uniformisées dans toute l'app.
- Typographie interface : **Manrope** (police principale). Le logo garde sa police historique, non touchée.
- Spec consolidée et faisant autorité : `docs/GROUPE RECA - 07 - CAHIER DES CHARGES.md` (les docs 01-06 sont les livrables d'agence d'origine dont il découle).

## Stack technique
- React 19 + Vite + TypeScript + React Router + TanStack Query + React Hook Form/Zod + Tailwind.
- Backend : Supabase (Postgres, Auth, Storage). Projet live `ynsuxctqsvusbgcudcno` — les migrations SONT appliquées en live (vérifié par sonde REST sur `/rest/v1/clients`, 2026-07-10), même si `docs/07` et le CLI local (`LegacyProjectNotLinkedError`) suggèrent le contraire — ne pas se fier à ces deux signaux pour juger l'état du schéma.
- `.input/supabase` contient la clé `service_role` du projet live en clair (gitignored) — ne jamais la committer ni l'afficher. Aucune credential de login admin app n'est stockée dans le repo ; les flows nécessitant une authentification ne peuvent pas être testés end-to-end sans que l'utilisateur fournisse des identifiants.

## Conventions de code (doc 07 §3)
- PascalCase composants, camelCase hooks, `*.service.ts` / `*.schema.ts`, pas de `any`, ~300 lignes max par fichier, tous les appels réseau passent par `services/`, tous les formulaires passent par RHF + Zod.
- "Un module ne dépend jamais directement d'un autre" = un module en aval peut importer un composant (ex: `FormModal`) d'un module en amont pour créer l'étape suivante du pipeline (ex: QuoteDetailPage → ClientFormModal), mais jamais l'inverse. Dépendances toujours vers l'avant dans le pipeline Leads→Quotes→Clients→Contrats→Factures→Paiements.
- Structure exacte de chaque module (`types/`, `schemas/`, `services/`, `hooks/`, `components/`, `pages/`) : voir les modules Leads/Clients déjà livrés comme gabarit vivant plutôt que de re-dériver depuis les docs.

- Cadrage carte (zoom/pan) du Wizard Contrats (tâche 12, 2026-07-17) : chaque sous-étape de "Analyse de la propriété" (Localiser/Délimiter) monte sa **propre** instance Mapbox GL — un cadrage manuel (zoom/pan) choisi par l'utilisateur ne survit donc PAS d'une sous-étape à l'autre par défaut, `MapViewportController` relance `fitBounds()` à chaque montage. - Mapbox GL Draw — sommets invisibles pendant le tracé (tâche 12, 2026-07-17) : le mode `draw_polygon` **standard** de `@mapbox/mapbox-gl-draw` (comportement documenté de la librairie, `src/modes/draw_polygon.js`) n'affiche que 2 sommets pendant un tracé actif (le 1er + l'avant-dernier commis), jamais les sommets intermédiaires — normal à ne pas reconfondre avec un bug applicatif si ça revient. Fix retenu : mode `draw_polygon` custom (`utils/drawPolygonAllVertices.ts`, spread du mode standard + `toDisplayFeatures` étendu) qui ajoute un point décoratif par sommet commis, avec un `meta` **custom** (jamais `'vertex'`/`'feature'`) pour ne pas être hit-testé par `isVertex`/`clickOnVertex` de Draw — sinon cliquer près d'un ancien sommet finirait le polygone prématurément à cet endroit. Piège à connaître pour tout futur mode Draw custom dans ce projet : `MapboxDraw.modes`/`MapboxDraw.lib` (dont `createVertex`) sont exposés à l'exécution ET typés par `@types/mapbox__mapbox-gl-draw` (`MapboxDraw.DrawCustomMode`), pas besoin d'importer les chemins internes du paquet (`src/lib/...`) qui ne sont pas dans le point d'entrée public.

- Sprint 008 — Analyse de la propriété V1, carte immersive (2026-07-13, branche `sprint008-analyse-propriete`) : sous-étape "Localiser" du Wizard Contrats enrichie (`fitBounds` auto, contour rouge RECA, masque extérieur assombri, panneau info 30%/carte 70%). Décisions et pièges Mapbox GL à connaître pour tout futur layer carto de ce module (ou d'un futur module cartographique) :
  - **Un composant qui ne reçoit `map` qu'une fois non-null (contrat `MapCanvas`/`onMapReady`, déjà en place depuis le sprint007) reçoit une carte dont l'événement `load` est déjà passé** — attendre un futur `map.once('load', ...)` dans un tel composant ne se déclenchera jamais. Écueil rencontré avec `map.isStyleLoaded()` qui peut renvoyer `false` juste après `load` (tuiles satellite encore en cours de chargement) : le code attendait alors un `load` qui n'arriverait plus, et les sources/layers n'étaient jamais ajoutés. **Solution : ajouter les sources/layers directement**, sans passer par `isStyleLoaded()`/`load` dans ce genre de composant enfant.
  - **Un polygone "à trou" (anneau extérieur + anneau intérieur inversé, technique standard pour un masque GeoJSON) peut échouer silencieusement à se rendre visuellement dans un layer `fill` Mapbox GL**, même quand la géométrie est logiquement correcte (confirmé via `map.queryRenderedFeatures()`, qui utilise son propre test point-dans-polygone indépendant de la tessellation GPU) — suspecté : cas dégénéré de la tessellation `earcut` (anneau extérieur bien plus grand que le trou, au zoom satellite ~20). **Solution robuste et éprouvée : remplacer le "donut" par plusieurs rectangles simples** (sans trou) couvrant les alentours du contour (`buildMaskRects` dans `propertyBoundary.ts`) — à réutiliser pour tout futur masque similaire plutôt que de retenter la technique du trou.
  - **`setState(fn)` où `fn` est une fonction est interprété par React comme un updater fonctionnel**, pas comme la nouvelle valeur d'état — piège rencontré avec `onReady={setRecenter}` (recenter une fonction), qui appelait silencieusement `setRecenter(prev => fn(prev))` au lieu de stocker `fn`. Solution : envelopper la feuille dans un conteneur `overflow:hidden` borné exactement à la zone visible, avec `pointer-events-none` sur ce conteneur et `pointer-events-auto` seulement sur la feuille elle-même (sinon le conteneur "vide" bloque aussi tout, transparent ou pas — un `div` sans fond visible n'est jamais transparent aux événements par défaut).

- Tâche 1 (2e occurrence de ce nom) — Détection automatique du stationnement par IA à l'étape Délimiter (2026-07-17, branche `optimisation-outil-de-mesure`) : première **Edge Function Supabase** du repo (`supabase/functions/analyze-satellite-image/`, Deno) — décisions à connaître pour toute future Edge Function de ce projet :
  - **La clé d'un fournisseur IA externe (ici `GEMINI_API_KEY`) ne doit jamais transiter par `VITE_*`/`.env`** (contrairement à `VITE_MAPBOX_TOKEN`, un token public conçu pour le navigateur) — elle vit uniquement comme secret Edge Function (`supabase secrets set`), lue via `Deno.env.get()` côté serveur. `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` sont auto-injectés par la plateforme Edge Function (pas à poser manuellement), utilisés ici pour télécharger l'image depuis le bucket privé `contract-captures` en contournant RLS.

Chaque correctif du code de l'Edge Function nécessite un nouveau `supabase functions deploy analyze-satellite-image` par l'utilisateur (2 redéploiements ont été nécessaires pour cette tâche, voir les 2 points suivants) — en attendant un déploiement, le bouton échoue proprement (toast d'erreur), sans rien casser.
